# Map Crisis Accountability Before Answering the Call

Use this prompt before agreeing to rescue a failing project or fix a system that generated the crisis. The article's argument: the pattern only changes when the cost lands where it belongs. If accountability does not land where the decisions were made, the pattern repeats.

Run this before you say yes — not after.

---

## Prompt

```
Here is the crisis I've been asked to fix: [DESCRIBE THE SITUATION — what failed, what the current state is, what is being asked of you, and by when].

Trace the decisions that created it: who made the calls that led to this point, when, and with what information? If I fix this, who learns the lesson — and who doesn't? Does fixing this change the conditions that produced it, or does it restore the conditions so they can produce it again?
```

---

**How to use:** Describe the crisis as factually as you can — the timeline, the decisions, the people involved. The prompt is not asking for blame allocation; it is asking for accountability mapping. There is a difference.

**The decision the article frames:** Answering the call is a choice. So is not answering it, or answering it with conditions that change where the cost lands. This prompt gives you the information to make that choice deliberately rather than by default. The quality of the rescue does not change the nature of the pattern — but where the accountability lands might.
